Applicazioni multi-container

Docker Compose è un tool che permette la definizione e l'esecuzione di applicazioni multi-container, può essere definito come un orchestratore di Dockerfile che però al contrario di Kubernetes e Swarm non può gestire host multipli.

Docker Compose viene utilizzato sempre più spesso dalle aziende sopratutto per l'esecuzione di app che necessitano di un back-end e di un front-end, per la realizzazione verranno quindi usati vari servizi, per esempio: Angular, Flask, MySQL

L'esecuzione richiede un file di configurazione chiamato: docker-compose.yml

File Compose YAML

version: "3.5"
services:
backend:
build:
context: ./backend
dockerfile: Dockerfile
ports:
- "3000:3000"
links:
- mongo
mongo:
image: mongo
volumes:
- ./db_vol/:/data/db
ports:
- "27018:27017"

Compose file

Tramite Docker Compose si possono creare tutti i servizi di una certa configurazione tramite un solo comando. La tecnologia di Compose realizza per noi in pochi secondi tutte le immagini necessarie per il funzionamento dell'applicazione.

Compose può essere anche usato per eseguire un singolo container mantendendo tutte le specifiche di esecuzione.

Dopo aver specificato i vari campi come nell'esempio si può procedere con l'avvio di Docker Compose, tramite il comando: $ docker-compose up

Per fermare Compose si esegue il comando: $ docker-compose down

Docker con Kubernetes

Kubernetes è un orchestratore che si occupa di distribuire i container attreaverso cluster di macchine fisiche o virtuali, automatizzando la distribuzione e la disponibilità di applicazioni basate su container.

L'abbinamento Docker-Kubernetes è attualmente la miglior soluzione per creare un ambiente di container solido ed efficiente, l'interazione tra i due è talmente forte che Docker può essere acquistato direttamente in combinazione con Kubernetes.

Se un server di cluster della rete sottostante dovesse cadere, Docker se ne accorgerebbe immediatamente e instaurerebbe nuovi container identici per fare in modo che il servizio non smetta di funzionare.

In seguito Kubernetes capirebbe in che modo e su quali altri server mandare i nuovi container; tenendo conto del carico di lavoro e della disponibilità dei server stessi.
